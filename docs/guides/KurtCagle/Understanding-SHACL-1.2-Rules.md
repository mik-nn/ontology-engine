In several earlier articles about the upcoming SHACL 1.2 release, I made a critical point - SHACL is a way to structurally describe a graph, rather than a way to describe it inferentially. What I meant by that is that when you create a SHACL node shape or property shape, you are defining how the graph itself is constructed, rather than ascribing logical definitions of class behaviour.

This distinction is important because the Web Ontology Language (OWL) is based primarily upon formal (assertional) logic. Indeed, OWL predates the Semantic Web by a few years and draws more on predicate logic than on graph theory. This is good in many respects: OWL lets you make inferences, which are conclusions based upon assertions. For instance, in OWL, it is possible to assert that the definition of "aunt" is the female sibling of a person's parent.

In earlier versions of SHACL, you could describe the predicates that a person has (`ex:hasSibling` and `ex:hasParent`, for instance), but SHACL tells you only structural information and doesn’t allow you to name relationships. This is why SHACL should be considered a structural rather than an inferential schema. By itself, this is also useful: it is a description of the graph rather than any particular meaning implied by the graph, which makes it easier to work with for validation and UI purposes. Still, it also means that there is less robustness in the model when you do need that inference.

## SPARQL and SHACL Rules

Before the advent of SPARQL, most interaction with knowledge graphs (though they weren’t called that at the time) assumed that they were static and that all assertions within the graph were valid. To use the knowledge graph, **rules** could be invoked (via SPIN or other early standards) to identify subgraphs and generate new assertions based on them, which are then maintained as virtual triples.

SPARQL was introduced in 2007 with two roles: generating tables by replacing variables with specific terms in the graph, and directly generating graphs via the CONSTRUCT statement. However, CONSTRUCT relied on human intervention to reintroduce new graphs generated by these queries into the system. SPARQL 1.1 Update squared the circle here by enabling the creation of new triples in a QUERY that are directly inserted into the original graph or a target name graph. While this enabled one-off updates, it didn’t provide a clean mechanism for updating specific properties.

The RDF/SPARQL/SHACL 1.2 standard (in progress) provides an updated version of the older rules architecture.

For instance, consider the following dataset.

```
# Data
ex:Bob ex:hasParent ex:Larry .
ex:Larry ex:hasSibling ex:Jane .
ex:Larry ex:hasSibling ex:Mike .
ex:Jane ex:hasGender ex:Female .
ex:Mike ex:hasGender ex:Male .
```

With SHACL 1.2, we can define the ex:hasAunt rule:

```
RULE {
    ?a ex:hasAunt ?c
    }
WHERE {
    ?a ex:hasParent ?b .
    ?b ex:hasSibling ?c .
    ?c ex:hasGender ex:Female .
}
```

This rule uses the [Shape Rules Language (SRL)](https://w3c.github.io/data-shapes/shacl12-rules/#shapes-rules-grammar), which is part of the SHACL 1.2 first working draft. It is intended to be run by a rules processor integrated with a triple store or by a command-line tool. The syntax can be expressed in several ways. For instance, the same rule can be given as:

```
 IF {
    ?a ex:hasParent ?b .
    ?b ex:hasSibling ?c .
    ?c ex:hasGender ex:Female .
}
THEN { ?a ex:hasAunt ?c }
```

or in Prolog format as

```
{ ?a ex:hasAunt ?c } :- {
    ?a ex:hasParent ?b .
    ?b ex:hasSibling ?c .
    ?c ex:hasGender ex:Female .
}
```

When this rule is run, it will find all graphs that satisfy the WHERE clause. In this case, the matching triples are as follows:

```
ex:Bob ex:hasParent ex:Larry .
ex:Larry ex:hasSibling ex:Jane .
ex:Jane ex:hasGender ex:Female .
```

Once the relevant triples are found, the SHACL engine will construct a new triple into the graph:

```
ex:Bob ex:hasAunt ex:Jane .
```

Note that the first SRL form is nearly identical to the SPARQL 1.1 Update statement:

```
INSERT {
    ?a ex:hasAunt ?c.
    }
WHERE {
    ?a ex:hasParent ?b .
    ?b ex:hasSibling ?c .
    ?c ex:hasGender ex:Female .
}
```

Given that, why the need for another construct? The distinction is architecturally significant and reflects different design philosophies.

### SHACL Rules: Declarative, Shape-Bound Inference

SHACL 1.2’s Rules section (building on SHACL-AF) provides _declarative inference_ that’s tightly coupled to the shape/target paradigm. Rules fire when focus nodes match shape targets, producing new triples via shape-conformance patterns. The key constructs are `sh:TripleRule` (for simple s-p-o generation) and `sh:SPARQLRule` (for more complex derivations).

The critical distinction: SHACL Rules are **monotonic** by design—they add triples to an inference graph but don’t delete. This makes them safe for layered reasoning, where you want to preserve the original data and clearly separate asserted from inferred statements.

### SPARQL UPDATE: Imperative Graph Mutation

SPARQL UPDATE is fundamentally imperative—it directly mutates the target graph store with INSERT, DELETE, and DELETE/INSERT operations. It operates on arbitrary WHERE patterns rather than being bound to shape targets, and it’s explicitly non-monotonic (deletions are first-class operations).

**Key Architectural Differences**

-   **Execution trigger:** SHACL Rules fire when focus nodes match shape targets; SPARQL UPDATE requires explicit invocation
    
-   **Monotonicity:** SHACL Rules are additive only; SPARQL UPDATE supports deletion
    
-   **Integration:** Rules are unified with validation in the same shape; UPDATE is a separate operation
    
-   **Ordering:** Rules use `sh:order` within shape context; UPDATE requires explicit query sequencing
    
-   **Result:** Rules typically write to a separate inference graph; UPDATE directly mutates the store
    
-   **Provenance:** Rules are naturally traceable to their triggering shape; UPDATE requires external tracking
    

**Why the Separation Matters**

For your SHACL 1.2 work, the rules section gives you a clean way to define _derived properties_ that are computed on-demand or materialised predictably—without the complexity of managing update semantics. You can declare “every instance matching this shape should also have these computed triples” as part of your schema definition, keeping inference logic co-located with validation constraints.

SPARQL UPDATE remains the right tool when you need true graph surgery—conditional deletions, data migrations, or ETL pipelines where you’re transforming rather than enriching.

## SHACL Rules Node Expression Bindings

It is also possible to create Rules using SHACL node expressions. This was touched upon in an earlier article. The Aunt rule would look something like the following:

```
ex:AuntRule
    a sh:TripleRule ;
    sh:subject sh:this ;
    sh:predicate ex:hasAunt ;
    sh:object [
        sh:filterShape [ sh:hasValue ex:Female ; sh:path ex:hasGender ] ;
        sh:nodes [ sh:path ( ex:hasParent ex:hasSibling ) ]
    ] .
```

The `sh:TripleRule` identifies the class of the shacl object, while `sh:subject`, `sh:predicat`e, and `sh:object` identify the triple being bound, with the `sh:this` providing the focus node (the specific entity that the rule applies to). In essence, this is constructing the triple:

```
ex:Bob ex:hasAunt ex:Jane .
```

The `sh:object` identifies a particular filter shape, here one where the `sh:path` of ex:hasGender is set to the value `ex:Female`. If this isn’t true, then the rule does no processing. The `sh:node`s then indicates the path to the relevant object if the filter shape is valid.

This approach is helpful if you are generating rules within the dataset (and may be the form in which such rules are actually encoded as RDF, though that’s not necessarily a given). a `sh:TripleRule` object is like a `sh:PropertyShape` object - it indicates a matching scope for a triple (rather than a property path) that can have metadata assigned to it as well as generating output. Put another way, it’s a template.

This does the same thing as the SRL form.

## SHACL SPARQL Rules

Finally, you can use the `sh:SPARQLRule` constructor:

```
@prefix sh:   <http://www.w3.org/ns/shacl#> .
@prefix ex:   <http://example.org/> .
@prefix xsd:  <http://www.w3.org/2001/XMLSchema#> .

ex:PersonShape
    a sh:NodeShape ;
    sh:targetClass ex:Person ;
    sh:rule ex:AuntRule, ex:UncleRule .

ex:AuntRule
    a sh:SPARQLRule ;
    sh:order 1 ;
    sh:construct """
        CONSTRUCT {
            $this ex:hasAunt ?aunt .
        }
        WHERE {
            $this ex:hasParent ?parent .
            ?parent ex:hasSibling ?aunt .
            ?aunt ex:hasGender ex:Female .
        }
    """ .

ex:UncleRule
    a sh:SPARQLRule ;
    sh:order 2 ;
    sh:construct """
        CONSTRUCT {
            $this ex:hasUncle ?uncle .
        }
        WHERE {
            $this ex:hasParent ?parent .
            ?parent ex:hasSibling ?uncle .
            ?uncle ex:hasGender ex:Male .
        }
    """ .
```

This binds two rules - the aunt rule and the uncle rule, to a given node shape, here one associated with the `ex:Person` class. This uses SPARQL syntax, with the $this variable holding the relevant focus node, the WHERE statement providing the context, and the CONSTRUCT statement providing the output.

This form provides a middle ground between SRL (available when the latter is not available) and the sometimes confusing node-expression form. These rules are triggered when a relevant focus node (a person) is detected, and the constructed triples are added to the output graph.

## Understanding Rule Graphs

It may not be obvious what is happening with triple production here (which is what all rules share). Unlike with SPARQL Update (which can write triples directly back into the graph), SHACL rules are in effect transformations. The SHACL is typically in its own graph, rather than mixed with the data graph. The SHACL rules engine reads the SHACL rules, then applies them to the data set. The output is then placed into a third graph (the target graph).

[

![](https://substackcdn.com/image/fetch/$s_!qaUO!,w_1456,c_limit,f_auto,q_auto:good,fl_progressive:steep/https%3A%2F%2Fsubstack-post-media.s3.amazonaws.com%2Fpublic%2Fimages%2Fc403337f-6794-4715-8c47-e8be5c6e3484_3715x856.png)

](https://substackcdn.com/image/fetch/$s_!qaUO!,f_auto,q_auto:good,fl_progressive:steep/https%3A%2F%2Fsubstack-post-media.s3.amazonaws.com%2Fpublic%2Fimages%2Fc403337f-6794-4715-8c47-e8be5c6e3484_3715x856.png)

This model is essential because it separates the source data triples from the target triples; you can query both together readily enough, but by keeping them separate this provides a way to create two graphs - a “deep” source graph that frequently keeps long-term records, and a shallower “now” graph that represents a view on that graph with constructed triples that represents a current state (and that is often much easier to query).

For instance, consider a situation where you have a person with a birthdate, but you only want to display their age. The input (data) graph may look like this:

```
ex:JaneDoe a ex:Person ;
    ex:name "Jane Doe" ;
    ex:dateOfBirth "2000-01-01"^^xs:date ;
    .
```

The SHACL ruleset can be given in SRL:

```
PREFIX ex:  <http://example.org/>
PREFIX xsd: <http://www.w3.org/2001/XMLSchema#>
PREFIX class: <http://example.org/ns/class#>
PREFIX person: <http://example.org/ns/person#>

RULE {?p a class:Person}
WHERE {
    ?p a ex:Person
}

RULE { ?p rdfs:label ?name }
WHERE {
    ?p a ex:Person ;
       ex:name ?name .
}

RULE { ?p person:ageInYears ?age }
WHERE {
    ?p a ex:Person ;
       ex:dateOfBirth ?dob .
    BIND(FLOOR(DAYS(NOW() - ?dob) / 365.2425) AS ?age)
}
```

SPARQL Rules:

```
@prefix sh:   <http://www.w3.org/ns/shacl#> .
@prefix ex:   <http://example.org/> .
@prefix xsd:  <http://www.w3.org/2001/XMLSchema#> .
@prefix class: <http://example.org/ns/class#>
@prefix person: <http://example.org/ns/person#>

ex:PersonShape
    a sh:NodeShape ;
    sh:targetClass ex:Person ;
    sh:rule ex:TypeRule, ex:DisplayNameRule, ex:AgeRule .

ex:TypeRule
    a sh:SPARQLRule ;
    sh:order 1 ;
    sh:construct """
        CONSTRUCT {
            $this a class:Person .
        }
        WHERE {
            $this s ex:Person .
        }
    """ .


ex:DisplayNameRule
    a sh:SPARQLRule ;
    sh:order 1 ;
    sh:construct """
        CONSTRUCT {
            $this rdfs:label ?name .
        }
        WHERE {
            $this ex:name ?name .
        }
    """ .

ex:AgeRule
    a sh:SPARQLRule ;
    sh:order 2 ;
    sh:construct """
        CONSTRUCT {
            $this person:ageInYears ?age .
        }
        WHERE {
            $this ex:dateOfBirth ?dob .
            BIND(FLOOR(DAYS(NOW() - ?dob) / 365.2425) AS ?age)
        }
    """ .
```

or Node Expressions:

```
@prefix sh:    <http://www.w3.org/ns/shacl#> .
@prefix ex:    <http://example.org/> .
@prefix xsd:   <http://www.w3.org/2001/XMLSchema#> .
@prefix shnex: <http://www.w3.org/ns/shacl-node-expr#> .
@prefix class: <http://example.org/ns/class#>
@prefix person: <http://example.org/ns/person#>

ex:PersonShape
    a sh:NodeShape ;
    sh:targetClass ex:Person ;
    sh:rule ex:TypeRule, ex:DisplayNameRule, ex:AgeRule .

ex:TypeRule
    a sh:TripleRule ;
    sh:order 1 ;
    sh:subject sh:this ;
    sh:predicate rdf:type ;
    sh:object [ sh:nodeValues class:Person ] .

ex:DisplayNameRule
    a sh:TripleRule ;
    sh:order 1 ;
    sh:subject sh:this ;
    sh:predicate rdfs:label ;
    sh:object [ sh:path ex:name ] .

ex:AgeRule
    a sh:TripleRule ;
    sh:order 2 ;
    sh:subject sh:this ;
    sh:predicate ex:ageInYears ;
    sh:object [
        shnex:floor [
            shnex:divide (
                [
                    shnex:days [
                        shnex:minus (
                            [ shnex:now () ]
                            [ sh:path ex:dateOfBirth ]
                        )
                    ]
                ]
                365.2425
            )
        ]
    ] .
```

The result in all cases will be the following (for December 31, 2025):

```
ex:JaneDoe a class:Person ;
    rdfs:label "Jane Doe" .
    person:ageInYears 25 .
```

For January 1, 2026, the output will, of course, magically change:

```
ex:JaneDoe a class:Person ;
    rdfs:label "Jane Doe" .
    person:ageInYears 26 .
```

which is a way of saying both “Happy Birthday” and “Happy New Year”!

_**One subtle point to note:**_ I deliberately changed the namespaces in the ruleset from `ex:` (the input) to `person:` and `class:` in the output, to illustrate how it can be done. This isn’t always necessary, but you are, in effect, creating a new graph with new properties in the target graph by applying rules.

## Recursion and Inferential Reasoning

One of the most significant issues in using SPARQL _without_ a reasoner is that you have to go to incredible lengths to handle subclasses and other transitive closures, which more than one person has told me was one of the key benefits of using RDF in the first place. In SPARQL, you have to create a fairly complex structure to iterate over all of the superclasses of a given class, as an example:

```
# SPARQL 
SELECT ?animal ?class WHERE {
    VALUES ?animal {ex:BrightEyes}
    ?animal a ?baseClass .
    ?baseClass rdfs:subClassOf* ?class .
    ?class rdfs:subClassOf* ex:Animal .
}
```

With SHACL Rules (here SRL) these triples can be rendered explicitly into the output graph:

```
PREFIX rdf:  <http://www.w3.org/1999/02/22-rdf-syntax-ns#>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>

# Instance propagation: if ?x is a member of ?class, 
# and ?class is a subclass of ?superclass, 
# then ?x is also a member of ?superclass
RULE { ?x rdf:type ?superclass }
WHERE {
    ?x rdf:type ?class .
    ?class rdfs:subClassOf ?superclass .
}

# Transitivity: subClassOf chains collapse
RULE { ?a rdfs:subClassOf ?c }
WHERE {
    ?a rdfs:subClassOf ?b .
    ?b rdfs:subClassOf ?c .
}
```

If you had the following source data:

```
ex:BrightEyes a ex:Cat.
ex:Cat rdfs:subClassOf ex:Feline .
ex:Feline rdfs:subClassOf ex:Mammals .
ex:Mammal rdfs:subClassOf ex:Vertebrate .
ex:Vertebrate rdfs:subClassOf ex:Animal .
```

Then the ruleset will generate the following output triples:

```
ex:BrightEyes a ex:Cat, ex:Feline, ex:Mammal, ex:Vertebrate, ex:Animal .
```

Because these are generated triples, it means that subsequent SPARQL queries against them to get a class interface and properties become much cleaner:

```
# SPARQL
prefix sh:    <http://www.w3.org/ns/shacl#> 
prefix ex:    <http://example.org/> 
prefix xsd:   <http://www.w3.org/2001/XMLSchema#> 
prefix mammal: <http://example.org/ns/mammal#>


SELECT ?animal ?nursesYoung WHERE {
     VALUE ?animal {ex:BrightEyes}
     ?animal a class .
     ?class mammal:nursesYoung ?nursesYoung .
     }
```

With the result:

```
{animal: ex:BrightEyes, nursesYoung: true }
```

This is true not because Bright Eyes is a cat, but because cats are mammals, and mammals are defined as animals that nurse their young.

This indicates the power of SHACL Rules - you can make these kinds of inferences not because this information is stored in the source graph directly, but rather because these triples can be materialised via rules into the target graph. As such, you can do a lot of OWL-related things quickly, because the materialised triples necessary to support them can be generated via SHACL. This also eliminates the need for static reasoners (often difficult to access) in triple stores, which is one reason I’ve not been a big fan of reasoners in the first place.

## The SHACL Rule Paradigm

There are several advantages to the approach of using SHACL 1.2 Rules and multiple graphs:

-   The underlying source graph remains unchanged, except for ingestion updates outside the scope of the rules.
    
-   You perform the (expensive) calculations only once (possibly via a scheduled CRON job), and any queries against the target graph only need to perform index lookups, as with any SPARQL query.
    
-   Rather than doing often complex insertions and deletions (as with SPARQL 1.1) you can wipe out the target graph (a fast operation) and write in a new one.
    
-   You can also maintain multiple target graphs so that updates to one graph don’t necessarily affect other target graphs.
    
-   Queries against the target graph are consequently much faster, easier to write, and potentially targeted at a specific audience.
    

Additionally, it should be evident that SHACL Rule Language (SRL) transformations are, in general, the easiest and cleanest to write for humans. However, from a machine (and AI) perspective, node expressions are likely the way to go. I suspect that future SHACL engines will likely transform one into the other internally, but that’s a design decision I have no control over.

SHACL 1.2 Rules provide the final piece in the SHACL framework. In many respects, they hold the same position in the RDF framework that XSLT or XQuery does in the XML world - a mechanism to transform one graph into another safely and declaratively.

It provides a layer of protection for the source graph; I think one reason SPARQL UPDATE never managed to gain much of a foothold was that there was just too much potential for mayhem to expose it to external users; SHACL provides a considerably better mechanism for separating working graphs from queryable graphs.

It also addresses many of the larger privacy issues: sensitive information is either redacted or never exposed, making SHACL 1.2 Rules a strong candidate for use cases ranging from banking to medical systems to personal data stores. All told, what is emerging is an intriguing and powerful set of tools for working with graphs intelligently and securely.
